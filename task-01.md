下面是你目前为止的**完整整体要求**（包含你后续补充的 GitHub 和 Skill 相关点），整理成一份可直接当「规格书」使用的版本：

---

## 一、总体目标

### 1）能力覆盖范围

需要调动全部可用能力，围绕以下方向持续、系统化地产出：

- 自动化办公（文档、表格、邮件、流程自动化）
- 自媒体（选题、写作、改写、分发）
- 数据分析（采集、清洗、分析、可视化、结论）
- 自动化提效（重复任务系统化、批处理、定时化）

信息来源上，必须包含多源内容，包括但不限于：RSS、博客、专栏、文章平台、**GitHub** 等。

### 2）结果标准

- 所有产出必须是 **真实、落地、可复用** 的场景与方法；
- 不能只停留在概念层面，必须有 **清晰的具体执行路径**；
- 要能形成 **每天可积累、可复盘、可迭代** 的成果体系，而不是一次性产物。

---

## 二、执行原则（工作方式）

1. **全流程由 AI 自主完成**：尽量不反复向你提问，由系统自动决策、推进。
2. 执行过程中以「**高价值产出**」为导向，减少无效动作和无意义抓取。
3. 优先筛选 **实操性强、见效快** 的案例、文章和项目。
4. 所有中间结果与最终结果都必须 **落盘保存**（写入文件），避免任何进度丢失。

---

## 三、文章处理要求（硬性）

### 1）每篇好文章 / 项目必须双产物

对每一篇被判定为“优质”的内容（包括文章、教程、GitHub 项目等），必须产出两份 Markdown 文件：

1. **原文版（original）**
   - 保留原文/项目的核心信息与结构；
   - 强调「可读、可追溯」：包括来源、链接、时间等；
   - 对内容做必要清洗和结构化（背景、问题、方案、步骤、效果、注意事项等）。

2. **公众号爆文改写版（wechat）**
   - 面向传播与阅读体验进行优化：
     - 标题更吸睛，有明确收益承诺（省时间、提效率、涨粉、提升转化等）；
     - 开头有强钩子：痛点共鸣、对比、小故事、典型场景等；
     - 结构重组，更易阅读：3–5 个清晰板块，“一屏一个重点”；
     - 内容尽量案例化，有具体步骤、情境；
     - 结尾有 **行动清单**，给出可执行步骤或模板提示。

### 2）文件命名要求

每一篇内容对应一对文件，命名规范严格统一：

- 原文结构版：`xxx-original.md`
- 公众号改写版：`xxx-wechat.md`

二者必须一一对应、成对出现。

## 四、Skill 产出要求（不仅整理，还要创新）

1. 不只是整理文章和项目，还要从中 **提炼“可技能化”的流程和套路**。
2. 需要新建 **可实际使用场景** 的 Skill 方案（不是停留在空想或抽象框架）。
3. 每个 Skill 至少要包含以下要素：
   - **解决什么问题**：用户的典型痛点、期望结果；
   - **触发条件**：在什么场景/时机触发这个 Skill（输入是什么、在哪用）；
   - **输入 / 输出**：输入参数或数据格式，输出是什么样的结果或状态；
   - **执行步骤**：细化到别人可以照着做，最好能够脚本化或模板化；
   - **适用边界与注意事项**：在什么场景不适用、容易踩哪些坑。
4. 最终目标是形成一个可以不断扩展和完善的 **Skill 库**，每天都有新增或迭代。

---

## 五、自动化与调度要求（硬性）

1. 设置定时任务（通过外部调度，如 cron/systemd 等实现），时间窗为：
   - 每日 **00:00 开始**
   - 每日 **08:00 结束**

2. 在该时段内，系统要持续执行完整流水线：

> 「搜集 → 筛选 → 保存 → 改写 → 沉淀 Skill」

3. 这一套夜间任务需要 **长期持续运行**，直到你明确叫停为止，而不是临时一次执行。

---

## 六、存储与目录要求（关键）

每天的结果必须 **分文件夹保存**，互不覆盖。统一目录结构：

根目录示例：

- `daily-results/YYYY-MM-DD/`

当日目录下固定结构：

- `original/`  
  存放当日所有原文结构版 Markdown：`*-original.md`
- `rewrite/`  
  存放当日所有公众号爆文改写版：`*-wechat.md`
- `skills/`  
  存放当日提炼出的新 Skill 草案/方案：`skill-*.md` 等
- `reports/`  
  存放当日总结与推荐清单，如：`summary.md`、`top-skills.md` 等
- `logs/`  
  存放执行日志、错误记录、过程数据，如运行日志、raw JSON 等

要求：**每日一个日期目录，执行多次也不互相覆盖**，同一日内文件命名和结构保持一致。

---

## 七、每日交付清单（你每天能看到什么）

每天至少能看到以下可视化成果（在当日目录下）：

1. **今日高价值 Skill 推荐清单**
   - 文件形式如：`reports/top-skills.md`
   - 包含若干重点 Skill + 各自的落地场景简述。

2. **今日文章原文与改写双文件**
   - `original/` 与 `rewrite/` 下，每篇内容都有 `original + wechat` 两个版本，命名一一对应。

3. **今日新增 Skill 设计/迭代内容**
   - `skills/` 目录下的新 Skill 文件，包含对于旧 Skill 的优化或全新 Skill 的设计。

4. **今日执行报告**
   - 如 `reports/summary.md`
   - 内容包括：今天做了什么、处理了多少文章/项目、产出多少 Skill、遇到什么问题、下一步建议。

## 八、质量要求（要有“惊喜感”）

1. 不仅完成基本任务，还要体现「超预期补充」，例如：
   - **组合流程建议**：将多个 Skill 串联成一个业务闭环（比如从素材收集、筛选，到排版发布）；
   - **跨工具联动玩法**：展示不同工具/平台之间的联动方案（如 飞书 + GitHub + 自动发布 等）；
   - **可直接复制使用的模板**：包括文案模板、操作 checklist、workflow 伪配置等，用户可以直接拿去用或稍改即用。

2. 每天的成果要体现出「**逐步进化**」：
   - 选题质量：越来越贴近实操和真实场景，减少空泛内容；
   - 改写效果：更适合传播、更直观、更易读；
   - Skill 完整度：结构更清晰，边界更明确，执行性更强。

## 九、GitHub 相关的特别要求（新增重点）

1. **GitHub 必须作为一个重要信息来源**，在其他内容源（RSS/博客/专栏等）基础上新增，不替代。
2. 在 GitHub 上重点关注的类型包括（不限于）：
   - 自动化办公相关脚本/项目（比如 Office 自动化、飞书/Slack/邮件自动处理等）；
   - 自媒体内容生产/分发相关工具（选题管理、内容 pipeline、定时发布等）；
   - 数据分析脚本/项目（采集、清洗、分析、可视化、报表自动生成等）；
   - 自动化提效相关 workflow（CI/CD、批处理任务、任务编排等）；
   - **已经封装成“Skill/模板”的仓库**：  
     - 如现成的 workflow 模板库、脚本工具集、自动化解决方案合集等；
     - 这些仓库需要被重点识别，并且在本地再进一步抽象为自己的 Skill（形成“从 GitHub Skill → 你的 Skill 库”的二次提炼）。

3. 语言方面 **没有偏好限制**：  
   - 中文、英文内容都可接受，只要实操价值高即可。
